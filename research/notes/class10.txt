Context Engineering and Agent Design for Research
The Unifying Idea
Over the past several weeks, you have been learning about something that the field has only recently given a name. When you designed prompts, and decided what information a language model should see and how it should be organized, when you built Claude Projects knowledge bases, and when you loaded papers into NotebookLM, you were creating persistent stores of information that the model could draw on across sessions. When you wrote CLAUDE.md and AGENTS.md files, you were encoding project-level conventions into a format that an agent reads at startup, every session, without being asked. When you built SKILL.md files and evaluated other people's skills, you were designing packages of on-demand context that load only when a specific task requires them. And when you chose to start a fresh Claude Code session rather than continuing a long one, or ran /compact at a natural task boundary rather than letting auto-compaction fire mid-debug, you were managing the contents of a finite resource under pressure.
All of these activities are instances of the same underlying discipline: context engineering. This class gives you the conceptual framework that ties them together, the empirical research that explains why it matters, and the practical habits that separate effective AI users from frustrated ones.
The term rose to prominence in mid-2025. In June, Tobi Lütke, the CEO of Shopify, described context engineering as[1] "the art of providing all the context for the task to be plausibly solvable by the LLM." Days later, Andrej Karpathy amplified the idea,[2] writing that people associate "prompts" with short task descriptions, but that in every serious LLM application, the real work is "the delicate art and science of filling the context window with just the right information for the next step." By September 2025, Anthropic had published a detailed engineering guide[3] defining context engineering as the set of strategies for "curating and maintaining the optimal set of tokens during LLM inference, including all the other information that may land there outside of the prompts." By early 2026, the concept had become the subject of peer-reviewed research, practitioner conferences, and its own growing body of best practices.[4] 
For you, as engineering researchers who will use AI tools for years to come, context engineering is arguably the most durable skill this course can teach. Models will change; interfaces will evolve; today's tools will be superseded. But the fundamental constraint—that a language model can only work with the information currently in its context window, and that the quality of its output depends on the quality of that information—is architectural. It comes from the transformer itself. Understanding how to work within that constraint, and how to design systems that manage it well, will remain relevant for as long as the transformer architecture (or anything like it) underpins the models you use.
From Prompt Engineering to Context Engineering
What Changed, and Why
In the early days of working with large language models—which, for most people, means late 2022 through 2023—the dominant skill was prompt engineering: the craft of writing instructions that reliably elicit useful outputs from a model. The basic concepts were: zero-shot and few-shot prompting, chain-of-thought reasoning, role-based personas, structured output formats. These techniques remain valuable; they are not obsolete. But they address only one component of what a model sees when it generates a response.
 Prompt-engineering terms:
  Zero-shot prompting: Asking the model simply to answer or do a task with no examples. 
  Few-shot prompting: Giving the model a few examples to show the pattern you want. 
  Chain-of-thought reasoning: Prompting the model to work through intermediate steps.
  Role-based personas: Telling the model to act like a specific role or expert to shape tone and style.
  Structured output formats: Requiring the response to follow a fixed schema or format, such as fields or JSON-like structure.
The shift from prompt engineering to context engineering reflects a change in how people use language models. When you type a question into a chatbot, the prompt is most of the context: your message, perhaps a short system instruction, and a few turns of conversation history. Prompt engineering is well-suited to this interaction pattern. But the moment you move beyond simple chat—the moment you use a Claude Project with uploaded documents, or run Claude Code with MCP servers connected, or invoke a skill that loads reference files on demand—the prompt becomes a small fraction of the total context. The system instruction, tool schemas, retrieved documents, conversation history, and injected memory all compete for the same finite space. What you write in the prompt may matter less than what surrounds it.
This is even more pronounced when agents are involved. A human user manages context naturally: you notice when a conversation has drifted, you scroll up to remind yourself of earlier details, you decide to start a new chat when the old one feels stale. An agent cannot do any of this unless it has been explicitly designed to. An agent running in a loop—reading files, calling tools, examining outputs, deciding what to do next—generates new context with every iteration. Each tool call returns results that enter the context window. Each reasoning step produces tokens that accumulate. Over a long task, the context fills with a mixture of critical information and incidental noise, and the model must attend to all of it when deciding the next action. If context management is not engineered into the system, the agent's performance degrades as the session progresses: not because the model has become less capable, but because the signal-to-noise ratio in its context has deteriorated.
This is the core insight: context engineering is not about writing better prompts; it is about designing the entire information environment in which a model operates. Prompts are one design surface. Tools, memory, retrieved documents, conversation history, and instruction files are others. The discipline is holistic.
The Attention Budget
Why does context quality matter so much? The answer lies in the transformer architecture you studied at the beginning of this course. Recall that in a transformer, every token can attend to every other token in the context. This is what gives the architecture its power: it can capture relationships between any two pieces of information, regardless of how far apart they are in the text. But it also means that the number of pairwise relationships the model must process grows quadratically with context length. For n tokens, there are n² pairwise attention computations. At 10,000 tokens, that is 100 million relationships. At 100,000 tokens, it is 10 billion.
As context grows, each individual token's share of the model's attention shrinks. Think of it as an attention budget: a finite resource that gets spread thinner as more tokens compete for it.[5] The model does not suddenly stop working at some threshold. Instead, its precision—its ability to locate and use specific pieces of information—degrades gradually. A model with 200,000 tokens of context capacity may exhibit measurable performance loss well before that capacity is reached, simply because the relevant information is diluted by irrelevant material. (The underlying mechanism is the softmax normalization in the transformer's attention layers: as more tokens compete for attention, each token's weight shrinks, reducing the model's ability to focus on any single piece of information.)
You encountered one manifestation of this previously: the lost-in-the-middle phenomenon, where information placed in the center of a long context is retrieved less reliably than information at the beginning or end.[6] At the time, we discussed this as a practical consideration for organizing your documents. Now you can understand why it happens: the model's attention patterns, shaped by training data distributions where shorter sequences are more common, allocate disproportionate weight to positional extremes. Longer contexts stretch these patterns beyond their training distribution, and the middle—the region farthest from both anchors— suffers most.
Recent research has formalized this degradation under the term context rot. In July 2025, the startup Chroma published a systematic study evaluating 18 frontier models—including GPT-4.1, Claude 4, and Gemini 2.5—on controlled tasks at varying context lengths.[7] Their central finding was stark: every model they tested exhibited performance degradation as input length increased, even on simple retrieval and text-replication tasks. The degradation was not uniform across models (some declined gracefully, others erratically), but it was universal. No model was immune.
Three mechanisms compound to produce context rot. First, the lost-in-the-middle effect: models attend well to the beginning and end of context but poorly to the middle, creating a blind spot that widens as context grows. Second, attention dilution: as the number of tokens increases, each token's attention weight decreases, and the model's ability to maintain focus on any single piece of information weakens. Third, distractor interference: semantically similar but irrelevant content actively misleads the model, pulling its attention toward plausible-but-wrong information. This third mechanism is particularly insidious in research contexts, where many documents in a literature collection share terminology and structure even when their specific findings differ.
The practical implication is a principle that should guide everything you do with AI tools from this point forward: more context is not better context. A model with a one-million-token context window does not benefit from having one million tokens loaded into it. It benefits from having the right tokens loaded—the minimum set of high-signal information needed for the current task, organized for efficient retrieval, with irrelevant material excluded rather than included "just in case."
The Components of Context
To engineer context effectively, it helps to have a vocabulary for its parts. Every time a language model generates a response, it is working with a context that can be decomposed into several components:[8]
Static context is information that remains constant across an entire session or deployment. System prompts are the most familiar example: the instructions that define the model's role, constraints, and behavior. CLAUDE.md and AGENTS.md files are static context: they load at session start and persist throughout. The system prompt you see in a Claude Project—the one you can edit in the project settings—is static context that applies to every conversation within that project.
Dynamic context is information that changes during a session. Conversation history is dynamic context: it grows with each exchange. Tool results are dynamic context: when an agent calls a web search tool or reads a file, the returned content enters the context window. Retrieved documents—whether pulled from a Claude Project knowledge base, a NotebookLM source collection, or a RAG pipeline—are dynamic context that appears when the system determines they are relevant to the current query.
Meta-context is the structural information that defines what the model can do, as distinct from what it knows. Tool definitions and MCP server schemas are meta-context: they tell the model which tools are available, what parameters each tool accepts, and what kind of results to expect. Each tool definition consumes tokens. An MCP server with ten tools, each with detailed parameter schemas, might add several thousand tokens of meta-context before the model has seen a single user message. If you have five MCP servers connected, the meta-context alone could consume 15,000–20,000 tokens.
These categories are not merely academic. Each one represents a design surface where you make decisions that affect the model's performance. When you write a CLAUDE.md file (static context), you are deciding what project-level information the agent should always have access to. When you choose which MCP servers to connect (meta-context), you are deciding how much of the model's attention budget to spend on tool definitions versus actual work. When you structure a skill with progressive disclosure—Level 1 description, Level 2 SKILL.md, Level 3 reference files (the architecture discussed in Class 9)—you are engineering dynamic context to load only what the current task requires.
The progressive disclosure system of skills is, in fact, a textbook example of context engineering applied to agent infrastructure. The single-line skill description (Level 1) sits in meta-context: it is always loaded, so the agent knows the skill exists and can decide whether to invoke it. The SKILL.md file (Level 2) loads only when the skill is triggered, adding task-specific instructions to the dynamic context. Reference files (Level 3) load only when the SKILL.md explicitly requests them, bringing in detailed domain knowledge for the specific sub-task at hand. At no point is all of this information in the context simultaneously. Each level loads just in time, keeps the context lean, and unloads when it is no longer needed. This is context engineering in action.
Context Hygiene in Practice
The previous section established the principle; this section makes it operational. If context is a finite resource with diminishing returns, then every token that enters the context window without contributing to the current task is waste. Not benign waste but active waste, because irrelevant tokens do not simply sit inert. They dilute attention, introduce potential distractors, and, in agent loops, compound across iterations. The discipline of keeping context lean and intentional can be called context hygiene.
What follows draws on a recent video by Nate Jones.[9] We previously shared his framework for how the skills standard has evolved, highlighting clear shifts since their introduction just six months ago. Here he addresses a complementary problem: how to avoid flooding agents (and your own interactive sessions) with information they do not need. The two concerns go together: a beautifully written SKILL.md is undermined if the agent's context window is already half-full of raw PDF metadata and stale conversation history by the time the skill loads.
This Matters More Than You Think
Jones opens his monologue with an economic argument that is worth internalizing even if you are not paying per token today. The frontier models you use through university-provided accounts like Gemini are subsidized, but they are not free to run. Rate limits are enforced because inference is expensive, and the cost scales with token volume: longer contexts mean more computation per call. When you hit a rate limit after 90 minutes of work, it is often not because you asked too many questions; it is because your accumulated context—conversation history, tool results, loaded documents, plugin schemas—grew so large that each exchange consumed a disproportionate share of your allocation.
This problem will only intensify. As next-generation models are deployed on more expensive hardware, the cost of frontier intelligence is projected to rise. Wasteful habits that are merely inconvenient today—hitting a rate limit, waiting for a slow response, watching output quality degrade mid-session—will become genuinely costly in professional settings. Jensen Huang, NVIDIA's CEO, has projected that individual engineers may eventually spend on the order of $250,000 per year on AI compute. Whether or not that specific figure proves accurate, the direction is clear: token efficiency is becoming a professional skill, not a curiosity.[10]
For you, the immediate motivation is simpler: better context hygiene means better outputs. A model working with 5,000 tokens of clean, relevant Markdown will outperform the same model working with 100,000 tokens of raw PDF bloat—not because the model is different, but because the signal-to-noise ratio is different. Everything said previously about attention budgets and context rot applies here. The practical question is: where does the waste come from, and how do you eliminate it?
Three Habits That Burn Your Context Budget
Format Discipline: Markdown over Raw Files
A common source of unnecessary token use is document format. When you drag a PDF file into a chat, the model does not see what you see—a cleanly typeset journal article with figures and equations. It receives a stream of extracted content that includes layout metadata, header and footer repetitions, font specifications, embedded formatting instructions, and structural artifacts that are meaningless to the model but consume tokens as surely as your actual text does.
Consider a typical journal article, say, a 15-page paper from the ASME Journal of Heat Transfer with the standard two-column format, headers, footers, page numbers, and a few figures. As a PDF ingested by a language model, this paper might expand to 25,000–30,000 tokens or more, depending on the extraction method and the complexity of the layout. The same paper converted to clean Markdown—body text, section headers, equations in LaTeX notation, figure captions retained, formatting artifacts stripped—might come to 5,000–8,000 tokens.[11] The information content is identical; the token cost differs by a factor of three to five.
The practical rule is straightforward: convert documents to Markdown before feeding them to a model. This applies to PDFs you upload to Claude Projects, papers you add to NotebookLM, and reference documents you include in a skill's reference files. Several tools can automate the conversion and the investment pays for itself immediately. If you are building a knowledge base of 50 papers for your literature review, the difference between raw PDFs and clean Markdown could be the difference between a functional knowledge base and one that exceeds the model's effective attention capacity.
 Light-weight document conversion tools:
  MarkItDown: Microsoft’s open-source, pip-installable converter for many file types; for PDFs it aims for straightforward text extraction. https://github.com/microsoft/markitdown/  
  pdf2md.morethan.io: A simple browser-based PDF-to-Markdown tool with drag-and-drop upload, good for quick one-off conversions with no install. https://pdf2md.morethan.io/  
  NoteGPT PDF to Markdown Converter: Free online tool that transforms PDF files into structured Markdown format, preserving elements like headings, lists, tables, paragraphs, and links. https://notegpt.io/pdf-to-markdown-converter 
Conversation Sprawl: Fresh Sessions over Marathon Chats
The second major source of waste is what Jones calls conversation sprawl: maintaining a single conversation for 20, 30, or 40 turns when the topic has shifted, the task has changed, or the conversation has simply accumulated too much history for the model to process effectively.
The mechanism is simple. Every time you send a new message, the model processes the entire preceding conversation as part of its context. Your first message, the model's first response, your second message, the second response—all of it re-enters the context window on every turn. By turn 30, the model may be processing 50,000 or more tokens of conversation history before it even begins to consider your latest question. Much of that history is irrelevant to the current task: you solved the bug in turn 8, discussed a different paper in turn 15, and now you are asking about figure formatting in turn 30. The model's attention is spread across all of it.
You have already encountered the consequence of this. In Class 7, we discussed Claude Code's /compact command, which summarizes earlier conversation to free up context space. Auto-compaction fires when the context approaches its limit, but it fires based on token count, not task boundaries, which means it can erase details the agent was actively using. The /compact command lets you take control by triggering compaction at a natural breakpoint. But compaction is a mitigation, not a solution; any summary is lossy, and information that seemed unimportant at the time of compaction may turn out to matter later.
The better habit is to avoid the need for compaction in the first place. Start fresh conversations at natural task boundaries. Finished debugging your simulation script? Start a new session before asking about your paper introduction. Completed a literature search? Start a new session before asking the model to synthesize findings. Jones recommends refreshing every 10–15 turns as a rule of thumb, but the deeper principle is to match session boundaries to task boundaries. Each conversation should have a clear purpose; when that purpose is fulfilled, the session has done its job.
A useful corollary: separate your research sessions from your execution sessions. Use one conversation to explore, gather information, and think through options. Then start a fresh chat—with a clean context—to execute the task, feeding in only the conclusions and decisions from the exploration phase, not the entire exploratory history. This mirrors a practice you already know from scientific work: you keep a lab notebook of everything you tried, but when you write the methods section, you include only what is needed to reproduce the final result.
The Plugin Tax: Auditing Your Meta-Context
The third source of waste is the one users are least likely to notice, because it is invisible by default. Every MCP server you connect, every plugin you enable, every tool you make available to the model loads its schema into the context window at the start of every call. These schemas—the JSON definitions that tell the model what each tool does, what parameters it accepts, and what it returns—are meta-context, and they consume tokens whether or not you use the tool in a given interaction.
A single complex MCP server might contribute 2,000–5,000 tokens of tool definitions. If you have five servers connected—say, GitHub, Google Drive, a web search provider, a database connector, and a browser automation tool—you could easily be spending 15,000–20,000 tokens on meta-context before you type your first message. Jones calls this the plugin tax: a silent overhead that many users never audit.
The recommendation from Anthropic's own documentation is to keep tool sets minimal and non-overlapping; if a human engineer cannot definitively say which tool should be used in a given situation, the model cannot be expected to do better.[12] OpenAI's guidance is even more specific: accuracy degrades noticeably past about 10 tool definitions, and they recommend fewer than 20 per agent.[13]
For your practical workflows, the rule is simple: connect only the tools you need for the current task, and disconnect the rest. If you are writing code, you probably need file access and terminal execution; you probably do not need a Google Calendar connector. If you are doing a literature review, you probably need web search, but you probably do not need a database tool. Auditing your connected tools before starting a session is a small habit with an outsized effect on the quality of your interactions.
Model Mixing as a Context Strategy
Budget forcing is the idea that different tasks warrant different models, and that using the most powerful (and expensive) model for every task is wasteful. Context engineering adds a new dimension to this idea. It is not just that smaller models are cheaper: the same prompt often performs better on a lighter model with clean, focused context than on a heavier model with a bloated context window.
The practical pattern Jones describes is a three-tier approach: use a reasoning model (such as Claude Opus or an o-series model) for complex planning, analysis, and decision-making; use a mid-tier model (such as Claude Sonnet) for execution tasks like writing code, drafting text, or processing data; and use a lightweight model (such as Claude Haiku) for formatting, proofreading, and cleanup. The key is matching the model's capacity to the task's requirements, and thereby keeping each model's context lean enough for it to do its best work.
You have already seen a version of this pattern in Claude Code's agent teams (Class 7), where the main Claude instance orchestrates sub-agents that may run on different models depending on the complexity of their assigned sub-task. The principle is the same whether you are dispatching sub-agents or simply choosing which model to use for your next conversation.
Five Principles for Agent-Level Context Design
The habits above apply to interactive use: your daily work with Claude, ChatGPT, or Gemini. But when you design agent workflows—when you write skills, configure pipelines, or build systems where AI operates with minimal supervision—the stakes are higher. An interactive user notices when the model's output degrades and can course-correct; an unsupervised agent cannot. Jones articulates five principles for agent-level context design that synthesize much of what you have learned so far: [14]
Index your references. Never provide an agent with a raw, complete document when it only needs specific information from that document. Instead, maintain an index—a structured list of what exists and where to find it—and let the agent retrieve only the relevant portions on demand. This is the same just-in-time retrieval pattern that Anthropic describes in their context engineering guide: agents that maintain lightweight identifiers (file paths, stored queries, document IDs) and load full content only when needed. It is also what Claude Code does when it searches a codebase: it does not load every file into context; it uses tools like grep and find to locate relevant files and reads only those.
Pre-process context for consumption. Data that enters an agent's context should arrive ready to use, not in need of cleaning or reformatting. If your agent will process simulation output, pre-convert it from raw solver output to a structured format the agent can parse efficiently. If it will review a manuscript, provide clean Markdown rather than a raw PDF. Every token the agent spends "reading" formatting artifacts or parsing messy input is a token not spent on reasoning about the actual task. This principle extends the format discipline discussed above from a personal habit to a system design requirement.
Cache stable context. In API-based workflows—which some of you may build as your research matures—system prompts, tool definitions, and reference documents that do not change between calls can be cached. Prompt caching allows the model provider to store these stable portions and reuse them across calls, avoiding the cost of re-processing identical content each time.[15] You are unlikely to implement prompt caching yourself in this course, but understanding that it exists explains something you have already observed: Claude Projects and system prompts are efficient because the platform caches them behind the scenes. When you invest time in writing a good CLAUDE.md file, the platform ensures that investment pays off across every session, not just the first one.
Scope context to the minimum. Do not give a planning agent access to a full codebase when it only needs an architecture overview. Do not give a formatting agent your entire project roadmap when it only needs the figure dimensions and journal style requirements. Each agent (or each step in a workflow) should receive only the information it needs for its specific role. This is one reason behind the allowed-tools field in the Agent Skills specification: by restricting which tools a skill can access, you prevent the agent from wandering into irrelevant territory and accumulating unnecessary context.
Instrument and measure. You cannot optimize what you do not observe. For agent workflows, this means tracking input tokens, output tokens, and cost per call. Which calls are expensive? Which ones produce the most waste? Where is the context window filling up with information that the agent never actually uses? These questions are answerable only if you have instrumentation in place. Even in interactive use, developing the habit of noticing when responses slow down, when output quality drops, or when you hit rate limits faster than expected gives you diagnostic information that can guide your context hygiene practices.
These five principles are not separate from the skill design principles you studied in Classes 8 and 9; they are the same principles restated at a different level of abstraction. A well-designed SKILL.md already embodies all five: it indexes references through the progressive disclosure system rather than dumping entire manuals into context, it specifies pre-processed output formats so downstream consumers receive clean data, it benefits from platform-level caching of its Level 1 description, it scopes its context to the minimum through allowed-tools and selective file loading, and it supports instrumentation through structured output that makes success and failure observable. When you write and evaluate skills, you are practicing agent-level context engineering. Now you have the vocabulary to articulate why those design choices matter.

Memory: Extending Context Across Time
Everything we have discussed so far operates within a single session. Context hygiene keeps the window clean for this conversation; attention budgets constrain what the model can attend to right now. But your research does not live in a single session. A PhD project spans years. A simulation campaign may run for months, with parameters evolving as you learn from each run. A paper revision cycle stretches across weeks of intermittent work: you draft a section on Monday, get feedback from your advisor on Thursday, and return to revise the following week. If context resets to zero every time you start a new conversation, you lose continuity and spend tokens re-establishing what the model already "knew" yesterday.
Memory is the set of mechanisms that allow context to persist beyond a single session. In one sense, you have been using memory systems since Week 2; you just have not called them that. In another sense, memory remains one of the least mature layers of the AI stack: a space where the tools are evolving rapidly and the best practices are still being discovered. This section introduces a framework for thinking about memory, maps the tools you already know onto that framework, and identifies what is coming next.
It helps to borrow a distinction from cognitive science. Human memory is not a single system; it comprises multiple systems that serve different functions.[16] The same is true for AI agents, though the implementations are mechanical rather than biological.
Conversation memory is the agent's awareness of what has happened in the current session. It is the most familiar type: the conversation history that the model re-reads on every turn. You manage it through the session discipline discussed above: starting fresh conversations at task boundaries, using /compact to summarize when a session runs long. Conversation memory is automatic (the platform maintains it) but ephemeral (it vanishes when the session ends) and lossy (compaction discards detail). It is the equivalent of working memory in human cognition: essential for the task at hand, limited in capacity, and not designed for long-term storage.
Semantic memory is persistent knowledge that the agent can access across sessions: facts, conventions, relationships, and domain expertise that remain relevant over time. Unlike conversation memory, semantic memory does not accumulate automatically; someone—you—must design and maintain it. What makes this category useful is recognizing how many implementations of semantic memory you have already built:
* Your Claude Project knowledge base is semantic memory. The papers, notes, and research proposals you uploaded persist across every conversation in the project. When you ask a question and the system retrieves relevant passages via RAG, it is drawing on semantic memory.
* Your NotebookLM source collections are semantic memory. The papers and notes you loaded for literature prioritization remain available across sessions, and the system retrieves from them whenever you ask a question.
* Your CLAUDE.md and AGENTS.md files are semantic memory. They encode project-level conventions—tech stack, coding standards, directory structure, physical unit systems—that the agent reads at the start of every session. The conjugate heat transfer example from Class 7 (SI units, NumPy docstrings, CSR sparse matrices, fluid-before-solid coupling order) is a compact semantic memory that tells the agent how to behave in this specific project, every time.
* Your SKILL.md files are a specialized form of semantic memory: task-specific knowledge that loads on demand rather than at startup. The writing standards skill built from Sainani's course is semantic memory about how to review scientific prose. The Level 3 reference files that a skill loads are semantic memory about domain-specific details.
What all of these share is that they are designed artifacts. Someone chose what to include, how to structure it, and where to store it. The quality of the semantic memory determines the quality of the agent's behavior across sessions. A Claude Project with 50 raw PDFs dumped in without curation is technically semantic memory, but it is poorly engineered semantic memory—the same distinction we drew before between wasteful and clean context, applied at a longer timescale.
Episodic memory is the agent's ability to learn from past interactions: not just what it knows, but what it has experienced. This is the newest and least mature category. Two implementations are worth knowing about.
First, Claude's memory feature on claude.ai generates memories from your conversation history—facts about your preferences, your project, your work patterns—and surfaces them in future conversations. If you tell Claude in March that you are working on conjugate heat transfer with FEniCS, it may recall that in April without being told again. This is episodic memory in a limited form: the system extracts facts from episodes (conversations) and stores them for future retrieval. The quality is variable; the memories are sometimes too specific, sometimes too vague, and they do not yet capture the kind of nuanced, evolving understanding that a human collaborator would develop over months.
Second, and more ambitiously, there is the idea of agents that write their own skills. When an agent notices that it has handled a task type repeatedly and successfully, it can extract the pattern into a new SKILL.md file, making that knowledge available for future sessions. Claude Code supports this through its skill-creator skill. The agent observes its own successful behavior, generalizes it, and persists it as a reusable artifact. This closes the loop between experience and knowledge, turning episodic memory into semantic memory. The quality of self-authored skills varies, and you would want to review and refine anything an agent produces before relying on it. But the direction matters: it points toward agents that improve not just within a session (through conversation memory) but across sessions (through accumulated, externalized knowledge).

Agent Design Patterns for Research
Everything in this class so far has been about making a single agent as effective as possible: engineering its context, managing its memory, keeping its attention budget lean. That emphasis is deliberate. Most individual-researcher workflows are best served by one well-configured agent, and the context engineering principles you have just learned are what make that agent capable enough to handle complex, multi-step tasks without needing reinforcements.
But some tasks genuinely benefit from distributing work across multiple agents or structuring the workflow in a way that goes beyond a single ReAct loop. To decide when that is the case, you need a vocabulary of agent design patterns: recurring architectures that match different task structures. You have encountered several of these already; this section names them and maps them to research workflows.[17]
Four Patterns Worth Knowing
Single agent with tools. This is the baseline, and it is more capable than most people assume. Claude Code with MCP servers connected, Claude with web search and file access, a Claude Project with a curated knowledge base: all of these are single-agent systems. The agent reasons, selects tools, acts, observes results, and iterates. What you have learned in this class about context hygiene, memory design, and attention budgets applies directly to making this pattern work well. A single agent with well-engineered context—a thoughtful CLAUDE.md, clean Markdown inputs, minimal tool sets, session boundaries at task transitions—can handle literature reviews, code debugging, data analysis, and manuscript drafting without any multi-agent coordination. Most of your research tasks should start here, and many should stay here.
Orchestrator-worker. One agent plans and delegates; workers execute specific sub-tasks and return results to the orchestrator for synthesis. You have seen this pattern twice already. In Class 7, Claude Code's /batch command decomposes a codebase-wide change into independent units, spawns a sub-agent for each unit in an isolated worktree, and aggregates their results. In Class 8, the academic-research-skills repository's academic-pipeline skill chains ten stages—research, write, review, revise—through an orchestrator that manages handoffs between specialized sub-agents. A more modest research example: a literature review workflow where the orchestrator decomposes a research question into search queries, dispatches them in parallel, collects results, and produces a synthesis. The orchestrator-worker pattern is justified when a task has genuinely independent sub-components that benefit from parallel execution.
Evaluator-optimizer. One agent produces output; a second agent evaluates it against criteria; the loop iterates until a quality threshold is met. This is what Claude Code already does internally when it writes code, runs tests, observes failures, and revises: the generation step and the evaluation step alternate until the tests pass. A research version of this pattern: draft a methods section, then have a second pass check it against journal guidelines and flag non-compliance, then revise and re-check. The evaluator-optimizer pattern is justified when you have clear, articulable quality criteria and when iterative refinement is cheaper than getting it right on the first attempt.[18]
Router. An initial classification step examines the incoming request and dispatches it to the appropriate specialized handler. Research example: a research assistant that handles "find papers on X" differently from "debug my simulation" differently from "format this figure for AIAA submission." Each handler can be a separate agent (or skill) optimized for its specific task type, with its own context, tools, and instructions. You have already worked with routing in a simpler form: the skill description functions as a routing signal (Class 9), telling the calling agent which skill to invoke for a given sub-task. Explicit router agents extend this idea to the workflow level.
The Practical Heuristic
Anthropic's guidance on agent design (footnote 17) makes a recommendation worth quoting: the most successful agent implementations use simple, composable patterns rather than complex frameworks. The advice is to start with the simplest solution that could work, and add complexity only when you have evidence that simplicity is not enough.
For your research, this translates to a decision rule. Default to a single agent with good context engineering. Escalate to an orchestrator-worker pattern when the task has genuinely independent sub-components and would benefit from parallelism, e.g., processing 50 papers concurrently, rather than sequentially. Use an evaluator-optimizer when you have explicit quality criteria that can be checked programmatically or by a second model pass. Use a router when your workflow must handle categorically different request types with specialized handlers.
What does not justify multi-agent complexity? Most single-researcher, single-task workflows. Interactive exploration sessions. Anything where the overhead of defining roles, managing handoffs, and debugging coordination exceeds the time you would spend doing it in one well-configured session. The coordination cost of multi-agent systems is high: shared state must be managed, context must be partitioned across agents (each with its own attention budget), and failures become harder to diagnose because the error could originate in any agent or in the handoff between them. A single capable agent with well-engineered context frequently outperforms a poorly coordinated team of agents, for the same reason that one senior researcher who knows the project deeply often outperforms a committee of people who each know only their slice.[19]
In sum: understand the patterns well enough to recognize when they would help; default to single-agent designs with good context engineering; revisit as the tooling matures. The context engineering principles from earlier in this class are not a prelude to multi-agent design; they are the design for most of what you will do.
Designing a Research Agent Workflow (In-Class Activity)
To close the class, you will apply what you have learned by sketching an agent workflow for a real task in your own research. This is not a polished deliverable; it is a thinking exercise that prepares you for Assignment 4, where you will write a CLAUDE.md for your research project. The CLAUDE.md is the semantic memory layer; this activity asks you to think one level up—from "what does the agent need to know about my project?" to "what does the agent need to do in my project, and how should the context be structured to support that?"
Your sketch should specify five things:
The task. Choose a concrete, recurring task from your research: running and post-processing a simulation, reviewing a draft, preparing figures for a submission, conducting a targeted literature search, debugging a solver. Be specific; "use AI for my research" is not a task.
The context the agent needs. What domain knowledge, data formats, naming conventions, and project-specific details must be present for the agent to do the task well? Which of these belong in always-on semantic memory (CLAUDE.md), which belong in on-demand memory (a skill), and which can be left to conversation memory?
The tools required. What MCP servers, file access, web search, or code execution capabilities does the agent need? Are there tools it should not have access to during this task, to keep its context lean?
The memory strategy. What information should persist across sessions (and where should it live), and what is session-level and can be discarded? If the task involves iterative refinement over multiple sessions—like a paper revision cycle—how does the agent recover state at the start of each new session?
The design pattern. Is this a single-agent task? Would it benefit from an evaluator-optimizer loop? Does it have independent sub-components that suggest an orchestrator-worker structure? Justify your choice based on the task structure, not on the desire to use something sophisticated.
Spend about 15 minutes sketching. Then we will have two or three volunteers share their sketches for class discussion. The feedback will focus on whether the context design matches the task requirements: is the sketch over-engineering the agent (adding complexity without justification), or under-engineering the context (leaving out information the agent will need)?
