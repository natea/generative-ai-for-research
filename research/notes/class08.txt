Agentic Coding Practicum — Building a Research Skill from Scratch
From Concepts to Practice
Class 7 covered a lot of ground. We traced the evolution from Copilot's autocomplete to terminal agents like Claude Code. We introduced the three-tier framework, MCP architecture, the skills system, CLAUDE.md and AGENTS.md conventions, and you installed your first terminal agents. That was one of the densest sessions in the course so far, and it was deliberately conceptual: you needed to understand what these systems are and how they work before using them on real problems.
This session is the other half of that equation. Today we build something. Specifically, we'll walk through a complete worked example: taking expert knowledge that exists as video lectures, distilling it into a structured standard, and packaging it as a Claude Code skill that can review any manuscript you're writing. By the end, you'll have seen every step of the process, from raw source material to a working SKILL.md file that changes how your agent behaves.
The example we'll use comes from a resource I've been recommending to my graduate students for over a decade: Dr. Kristin Sainani's "Writing in the Sciences" course.[1] Sainani is a professor at Stanford, she holds an MS in statistics and a PhD in epidemiology, and also trained in science writing at UC Santa Cruz. She has taught statistics and writing at Stanford for more than a decade and received the 2018 Biosciences Award for Excellence in Graduate Teaching. Her Coursera course, first released around 2012, teaches scientists to write with clarity and precision using practical examples and exercises. It covers principles of effective writing (cutting clutter, using strong verbs, active voice), sentence and paragraph architecture, the format of scientific manuscripts, peer review, and communicating science to broader audiences. The lectures are still available on YouTube, and I maintain a playlist[2] with all 30 videos that I share with every new student in my group. The videos are posted under a Creative Commons Attribution license (CC-BY), which allows any form of reuse and derivative works.
Here’s a thought: how might you take the wisdom in 30 video lectures and turn it into something that an AI agent can apply automatically when reviewing your manuscript drafts? That's what I did, and the process itself illustrates several key ideas from Class 7 in action.
The Workflow: From Videos to a Reusable Agent Skill
Here's the workflow that enabled this idea, step by step.
Step 1: Extracting the video URLs with Comet. I started with the YouTube playlist: 30 videos from lecture screen-casting recordings, each covering a different aspect of scientific writing. To feed these into an AI tool, I first needed the individual video URLs. Getting each one manually involves clicking a menu, then a Share option, then selecting and copying the URL, and pasting it elsewhere to form a list. So I opened the playlist in the agentic browser Comet, by Perplexity, and entered the following instruction:
"This YouTube playlist has 30 videos. I want you to click through the three-dots menu for each one, click Share, and copy the URL to the video, and collect the video URLs in a list, separated by a new line. Bonus points if you remove the tracking code, the thing that starts `?si=` followed by an alphanumeric."
In less than a minute, the agent gave me a clean, text-only list of all 30 URLs with the tracking parameters stripped out, ready to copy with a single button click. This is a mundane task, but notice the time savings: doing this manually (click, click, copy, paste, trim the URL, repeat 30 times) would take maybe 15–20 minutes of mind-numbing work. The agent did it in under a minute, and it didn't make a single copy error.
Step 2: Synthesizing the content with NotebookLM. I pasted all 30 URLs into a new Google NotebookLM notebook in one go. NotebookLM ingested the video transcripts as sources.[3] In the Studio, I requested a Report using specific instructions:
"Establish a unified standard for academic writing that prioritizes clarity, removes linguistic interference, and optimizes the structural flow of research manuscripts. Write the result in a Markdown file that is in the format of a SKILLS.md file to use with an AI agent that reviews a manuscript to detect any issues that need editing for complying to the guidelines."
NotebookLM produced a Writing Standards Manual: a comprehensive narrative document that distilled the key principles from all 30 lectures into a structured reference. It captured the core ideas: strip every sentence to its cleanest components, use active voice, resurrect smothered verbs, maintain keyword consistency, audit for numerical accuracy and citation integrity. However, the output was in the style of a narrative guideline—readable by a human, but not formatted as a skill that Claude Code could operationally follow. NotebookLM did its job: it synthesized the source material into a coherent document. But it doesn't know what a SKILL.md file needs to look like.
Step 3: Converting the guideline into a proper SKILL.md file with Claude. I pasted the full text of NotebookLM's Writing Standards Manual into a Claude conversation and asked:
"I have pasted a Writing Standards Manual that I produced based on the contents of the 'Writing in the Sciences' course by Prof. Kristin Sainani at Stanford. Create a SKILLS.md file on the basis of this, which can be dropped into the appropriate folder of a local repository for a working manuscript, and give detailed instructions on how to use the skill with Claude Code open in the terminal of VS Code to perform a writing-quality review of one's own manuscript."
Claude began by checking the skill format conventions. It looked at how existing skills are structured: the YAML frontmatter, the Markdown body, the relationship between the description field and the skill's trigger behavior. Then it produced two files: the SKILL.md itself, architected as an operational protocol with five sequential audit passes and four review modes, and a companion HOW-TO-USE.md document with installation instructions and example prompts.
The result: a SKILL.md file that, when placed in .claude/skills/manuscript-review/ inside any manuscript project, teaches Claude Code to perform a structured writing-quality review following the Sainani methodology. When you open Claude Code in that project and say "review my manuscript," it runs five passes—clutter extraction, voice and verb vitality, sentence architecture, keyword consistency, and numerical/citation integrity—and produces a structured report with specific findings, concrete revisions, and severity tags.
Three different AI tools used in this workflow, each doing what it does best. The browser agent handled the mechanical extraction. NotebookLM handled the synthesis of 30 video transcripts into a coherent document. Claude handled the architectural work of converting a human-readable guideline into a machine-actionable skill with proper formatting. No single tool could have done all three steps well.
What Makes a Skill a Skill: The SKILL.md Conventions
Let's now look carefully at what Claude Code expects when it discovers and loads a skill. Understanding these conventions is the prerequisite for creating your own skills (which is what Assignment 4 will ask you to do).
Where skills live
Claude Code looks for skills in specific directories. There are two levels:
Project-level skills live inside your project's .claude/skills/ directory. Each skill gets its own subfolder containing at least a SKILL.md file:
my-manuscript/
├── .claude/
│   └── skills/
│       └── manuscript-review/
│           └── SKILL.md
├── draft.tex
├── figures/
└── references.bib
Project-level skills are only active when Claude Code is running inside that project directory. This is the right choice for skills that are specific to a particular manuscript, codebase, or experiment.
Global skills live in your home directory at ~/.claude/skills/. Same subfolder structure, but the skill is available in every project you open with Claude Code. This is the right choice for skills you want everywhere, like a general writing review standard or your lab's figure formatting conventions.
There is also a third option if you're using Claude's web interface rather than Claude Code: you can upload SKILL.md files directly to a Claude Project's knowledge base via the Project Knowledge panel. This doesn't give you the full terminal agent experience, but the instructions in the SKILL.md still shape Claude's behavior within that project's conversations. This is how the academic-research-skills repository (which we'll look at in a moment) recommends using its skills on claude.ai.
The anatomy of a SKILL.md file
A SKILL.md file has two parts: YAML frontmatter and a Markdown body.
The frontmatter is the metadata block at the top of the file, delimited by triple dashes. It must contain at least two fields:
---
name: manuscript-writing-review
description: >
  Use this skill when asked to review, edit, or improve the writing
  quality of a scientific or engineering manuscript. Triggers include:
  "review my writing," "check my manuscript," "improve clarity,"
  "clean up the prose," "edit for journal submission"...
---
The name field is a short identifier in lowercase, hyphenated, no spaces. Think of it as the skill's internal label.
The description field is the critical one. This is what Claude Code reads to decide whether to load the skill for a given task. It functions as a trigger specification: when you ask Claude Code to do something, it compares your request against the descriptions of all available skills and loads the ones that match. This means the description needs to be written carefully: it should include the kinds of phrases a user would actually say when they want this skill applied. Notice in our example it lists specific trigger phrases ("review my writing," "check my manuscript," "improve clarity") and also specifies what the skill does not cover ("Do NOT use for content/technical review of methods or results"). Both positive and negative triggers help Claude Code make accurate matching decisions.
Progressive disclosure: how skills are loaded
Here's an important design principle. Claude Code does not load all skills into its context window at once. That would waste tokens, a precious and limited resource. Instead, it uses a two-stage process:
1. At startup, Claude Code reads only the YAML frontmatter (names and descriptions) of all available skills. This is lightweight: a few hundred tokens total even if you have dozens of skills.
2. When you make a request, Claude Code matches your intent against those descriptions. If it determines a skill is relevant, it loads the full Markdown body of that SKILL.md into context at that point. If no skill matches, none are loaded.
This is called progressive disclosure, and it's the same pattern used throughout modern AI systems: load the minimum information needed to make a routing decision, then load the full detail only when it's needed. It means you can have many skills installed without paying a token cost for the ones you're not using in a given session.
It also means the Markdown body—the bulk of the SKILL.md—is where you put all the detailed instructions, rules, examples, and reference tables. Be as thorough as you want here. This content only enters the context window when the skill is actually activated, so length is not a concern the way it would be if everything were loaded upfront.
What goes in the Markdown body
The body of the SKILL.md is where you encode the actual expertise. There's no rigid schema; Claude Code reads it as Markdown and follows the instructions it contains. But effective skills tend to share several elements:
A purpose statement. One or two sentences explaining what the skill does and what perspective the agent should adopt. In our example: "You are an expert scientific writing reviewer. Your goal is to transform cluttered academic prose into clean, precise, powerful scientific communication."
Review modes or operating modes. If the skill can be applied in different ways (full review vs. targeted pass vs. interactive editing), define them explicitly with a table mapping trigger phrases to behaviors. This helps the agent choose the right mode based on how the user phrases their request.
The core protocol. This is the substance: the rules, checklists, lookup tables, and procedures that encode the expertise. In our writing review skill, this takes the form of five sequential audit passes, each with specific patterns to look for, concrete replacements, and worked examples. For a figure-formatting skill, this might be specific dimensions, fonts, color palettes, and DPI requirements. For a simulation postprocessing skill, it might be the sequence of steps for extracting, cleaning, and visualizing results.
Output format specification. Tell the agent exactly how to structure its response. If you want a report with headings and severity tags, specify that. If you want before/after pairs with rationales, specify that. Agents follow explicit formatting instructions much more reliably than implicit expectations.
Constraints and boundaries. What the skill should not do is as important as what it should do. Our skill includes: "Never alter scientific content," "Respect disciplinary conventions," "Preserve the author's voice," "Be specific—every suggestion must include the original text and a concrete revision." These guardrails prevent the agent from overstepping its role.
Skills vs. CLAUDE.md: different roles, complementary functions
Recall from Class 7 that CLAUDE.md (or AGENTS.md) is the project-level instruction file that tells Claude Code about your codebase, conventions, and context. You might wonder how that differs from a skill.
The distinction is straightforward. CLAUDE.md describes your project, what it does, what tools it uses, how it's organized, what conventions to follow. It's read at the start of every session, unconditionally. It's like the onboarding document for a new collaborator in that project.
A skill describes a workflow or domain expertise, or how to perform a specific task well. It's loaded only when relevant. It's like the specialized knowledge a consultant brings for a particular job.
They compose naturally. Your CLAUDE.md might say: "This project is a manuscript for the Journal of Fluid Mechanics. The draft is in paper/ as LaTeX files. We use BibTeX for references." The manuscript-review skill adds: "When asked to review writing quality, apply these five audit passes in sequence..." The CLAUDE.md file gives the agent project context; the skill gives it task expertise. Together, they produce a review that respects your journal's conventions, knows where your files are, and applies rigorous writing standards.
Looking at a Skill in the Wild: The Academic Research Skills Repository
To give you a sense of how far the skills concept can go, let's briefly look at an open-source project that packages an entire academic research workflow as Claude Code skills: the "Academic Research Skills" repository by Cheng-I Wu (https://github.com/Imbad0202/academic-research-skills).
This project contains four interconnected skills:
* deep-research—a 13-agent research pipeline with modes for full investigation, literature review, fact-checking, systematic review with PRISMA, and Socratic guided research.
* academic-paper—a 12-agent paper-writing pipeline with LaTeX output, citation conversion across formats (APA, IEEE, Chicago, MLA, Vancouver), visualization, and revision coaching.
* academic-paper-reviewer—a 7-agent multi-perspective peer review system with 0–100 quality rubrics, including an Editor-in-Chief, three domain reviewers, and a Devil's Advocate.
* academic-pipeline—a 10-stage orchestrator that chains all of the above into a full research-to-publication pipeline: research, write, integrity check, review, Socratic coaching, revise, re-review, re-revise, final integrity check, and finalize.
The full pipeline can consume over 200,000 input tokens in a single run and requires Claude's most powerful model (Opus 4.6) on the Max plan. This is emphatically not something you'd run casually; it's an example of what becomes possible when you push the skills concept to its limit.
What I want you to notice is not the scale, but the structure. Each of these skills is a SKILL.md file following the same conventions we just discussed: YAML frontmatter with a name and trigger description, followed by a Markdown body with modes, agent roles, protocols, output formats, and constraints. The deep-research SKILL.md defines seven modes and thirteen agent roles. The pipeline SKILL.md defines ten stages with checkpoint behaviors. But the underlying format—the contract between the skill author and Claude Code—is identical to our 170-line writing review skill.
This is a general pattern in engineering: simple interfaces that scale. The SKILL.md format is simple enough that you can create a useful skill in an afternoon (as we just did), and powerful enough that someone can build a multi-agent research pipeline with the same mechanism.
For your purposes as engineering researchers, the sweet spot is much closer to what we built today than to the full pipeline. A skill that encodes your lab's figure formatting standards. A skill that knows how to postprocess your simulation output. A skill that applies your target journal's writing conventions. These are 50–150 line files that save you hours of repetitive work and ensure consistency across everything you produce. In Assignment 4, you'll create something at this scale for your own research—but we'll talk about that assignment separately.
Exercises: Putting the Agentic Coding Loop into Practice
The worked example above showed you the full arc of building a skill, from source material to a working SKILL.md. But skills are only one part of what makes agentic coding useful for research. The more fundamental capability is the interaction loop itself: describing a goal to the agent, reviewing what it produces, refining your instructions, and iterating until the result meets your standards. That loop is the core skill (the human skill, not the AI agent kind) that you need to develop.
Below are four exercises, ordered roughly from most immediately practical to most forward-looking. You don't need to complete all four—pick the one that's closest to a real problem in your current research, and spend your time there. Each exercise is designed to be achievable in a single session with Claude Code (or Copilot CLI, if you prefer), and each one teaches a different aspect of working with a terminal agent.
Exercise 1: Reproduce a Figure from a Published Paper
Take a figure from a published paper in your field—ideally one from your own research group, so you have access to the underlying data, or one where the data is available in the paper's tables or supplementary materials. Your task: starting from an empty project folder, use Claude Code to write a Python script that reproduces the figure to publication quality.
Start by creating a CLAUDE.md for the project. It doesn't need to be long, just describe the goal ("reproduce Figure 3 from [citation]"), the data source (a CSV file you provide, or hardcoded arrays from the paper's tables), and any formatting requirements from the target journal (column width, font sizes, file format). Then launch Claude Code and describe the figure you want.
The interesting part is the iteration. The first version will be functionally correct but aesthetically generic. This is where you learn to direct the agent: "Make the axes labels 9-point Times New Roman," "Switch to a two-panel layout with shared x-axis," "Use the same color scheme as the original—blue for experimental, red dashed for numerical," "Add error bars from the uncertainty column in the data file," "The legend is overlapping the data—move it outside the plot area." Each instruction teaches you something about how precisely you need to communicate visual intent to the agent, and how well it interprets ambiguous requests.
Tips for success:
* Prepare your data file before starting. The agent works much better when it can read an actual CSV or NumPy array than when it has to guess at numbers from a screenshot.
* Be specific about the target journal's figure requirements. If you know the column width (e.g., 3.5 inches for a single-column figure in an AIAA journal), font family, and required file format (EPS, PDF, high-resolution PNG), put that in your CLAUDE.md. The agent will use it.
* Don't accept the first draft. The pedagogical value is in the refinement cycle. Push for at least three or four rounds of revision. You'll discover which aspects of figure formatting the agent handles well (axis labels, legends, line styles) and which require more explicit direction (subplot spacing, inset plots, annotation placement).
* If the agent makes a mistake—wrong units on an axis, misinterpreted data columns—note how you catch it and how you communicate the correction. That's the "agentic engineering" judgment we discussed in Class 7.
Exercise 2: From Pseudocode to a Working Numerical Solver
Choose a numerical method you understand well, e.g., a finite difference scheme for the 1D heat equation, a fourth-order Runge-Kutta integrator for an ODE, a simple panel method for potential flow, a Newton-Raphson root finder, etc.. Write the pseudocode by hand (or pull it from a textbook) and save it as a Markdown file in a new project folder.
Create a CLAUDE.md that describes the project context: "This project implements numerical methods for [topic]. We use Python with NumPy and Matplotlib. All variables should use SI units. The code should include docstrings explaining the mathematical formulation." Then ask Claude Code to turn your pseudocode into a working implementation.
Once the basic solver works, extend it: ask the agent to add a grid convergence study that runs the solver at several resolutions and plots the error versus grid spacing on a log-log scale. Ask it to overlay the numerical solution against an analytical solution (if one exists for your test case). Ask it to refactor the single-file script into a module with separate functions for setup, time-stepping, and visualization.
The unique value of this exercise is verification. Unlike a generic coding task, you know the physics. You can look at the output and immediately tell whether the agent got the boundary conditions right, whether the convergence rate matches the expected order of accuracy, or whether the solution is physically plausible. This gives you ground truth for calibrating your trust in the agent's output, which is the single most important judgment skill for agentic coding.
Tips for success:
* Start with a problem that has an analytical solution. Being able to compute the exact error is worth more pedagogically than a more impressive-looking problem where you can only eyeball the results.
* Deliberately introduce an error at some point. For example, tell the agent "I think the boundary condition should be Neumann, not Dirichlet" when you know Dirichlet is correct, or change a coefficient in the pseudocode. See whether the agent catches the inconsistency or silently propagates it. This is the most instructive moment in the exercise: understanding what the agent does and does not verify on its own.
* Pay attention to how the agent handles mathematical notation. Does it preserve the variable names from your pseudocode? Does it document which equation is being discretized? These details matter for research code that you or a collaborator will need to understand months later.
Exercise 3: Refactor a Messy Research Script
This may be an immediately useful exercise. Find a script from your own research that you know is messy, the kind that works but that you'd be embarrassed to share with a collaborator. The symptoms are familiar: no docstrings, single-letter variable names, hardcoded file paths, a 200-line main function that does everything, results printed to the terminal instead of saved to a file, no error handling, maybe some commented-out blocks of code that "might be needed later."
Copy it into a new project folder. Create a CLAUDE.md that describes what the script does (what data it processes, what output it produces, what libraries it depends on). Then ask Claude Code to refactor it, one step at a time.
Start with the low-hanging fruit: "Add NumPy-style docstrings to every function." Then: "Replace all single-letter variable names with descriptive names that reflect the physics." Then: "Extract the plotting code into its own function." Then: "Replace hardcoded file paths with command-line arguments using argparse." Then: "Add a requirements.txt." Then: "Write a basic test that verifies the main function produces the expected output on a known input."
Each step is a self-contained, reviewable change. You'll see the agent editing multiple locations in the file to maintain consistency (when it renames a variable, it finds all usages). You'll see it making structural decisions (how to decompose a monolithic function, what to name the extracted functions). And you'll occasionally see it make mistakes: renaming a variable inconsistently, changing the logic slightly during a refactor, or introducing a subtle bug when extracting a function. Catching these is the exercise.
Tips for success:
* Use version control. Initialize a git repository before you start, and commit after each refactoring step. This gives you a clean diff to review at each stage, and it lets you roll back if the agent introduces a regression. Claude Code works naturally with git: you can ask it to commit with a descriptive message after each change.
* Review every diff carefully. The refactoring steps that look safe (renaming variables, adding docstrings) are exactly the ones where you might let your guard down. But a variable rename that misses one usage, or a docstring that describes the wrong behavior, can be a subtle source of bugs.
* After the refactoring is complete, run the script on your actual data and verify that the output is identical to what the original messy script produced. Refactoring should change structure, never behavior. If the outputs differ, you've found a bug the agent introduced, and debugging it together with the agent is a valuable learning experience.
* If your script is too long for a single session (more than a few hundred lines), focus on one section. You'll learn just as much from refactoring a single 80-line function as from attempting to clean up an entire 1,000-line pipeline.
Exercise 4: Build a Simple Custom Skill
This exercise looks forward to Assignment 4, where you'll build a CLAUDE.md for your own research project. Here, the task is smaller and more focused: create a single SKILL.md that encodes one specific piece of domain expertise relevant to your work.
Some concrete ideas to get you started:
* Journal figure formatter. Your target journal has specific requirements: figure width, font family and size, DPI, acceptable file formats, color accessibility guidelines. Encode all of this in a skill, so that any time you ask Claude Code to generate a plot, it automatically follows the journal's specifications. Test it by asking the agent to make the same plot with and without the skill loaded; the difference should be immediately visible.
* Simulation postprocessor. If your research involves running simulations (CFD, FEA, molecular dynamics), you likely have a standard sequence of steps for extracting results: read the output files, compute derived quantities (Nusselt number, drag coefficient, stress intensity factor), generate standard plots, write a summary. Encode that sequence in a skill.
* Data cleaning protocol. If you work with experimental data, you probably have a set of standard cleaning steps: remove outliers beyond a certain threshold, apply a moving average filter, resample to a uniform time base, or convert units. A skill that encodes these steps means the agent applies them consistently every time, and you don't have to repeat the instructions.
* Lab report template. If your group writes weekly or monthly progress reports with a standard structure (objective, methods, results, next steps), a skill can ensure that every report follows the template and that the sections are populated with the right level of detail.
Whichever topic you choose, the process is the same: write the YAML frontmatter with a clear name and a description that includes the trigger phrases you'd naturally use, then write the Markdown body with the specific rules, examples, and output format you want the agent to follow. Drop it in .claude/skills/your-skill-name/SKILL.md, start Claude Code, and test it.
Tips for success:
* Start with the description field. Before writing the full body, get the triggers right. Think about the exact phrases you'd use when you want this skill applied, and also think about the phrases that should not trigger it. A figure-formatting skill should activate when you say "make a plot" but probably not when you say "read the data from the figure in this PDF."
* Be concrete in the body. The difference between a useful skill and a vague one is specificity. "Use appropriate fonts" is vague. "Use 9-point Times New Roman for axis labels, 10-point bold for axis titles, 8-point for tick labels, and 7-point for annotations" is a skill the agent can actually follow.
* Include at least one worked example: a "before and after" showing what the agent's output should look like when the skill is applied correctly versus when it isn't. Examples are the single most effective way to communicate expectations to an LLM.
* Test the skill against a task you've done manually before, so you can compare the agent's output to your known-good result. If the skill produces a figure that doesn't match your journal's requirements, the problem is in your SKILL.md, not in the agent; refine the instructions and try again.
* Keep it short. A good first skill is 50–100 lines. You can always add detail later as you discover edge cases. The academic-research-skills repository we looked at earlier has SKILL.md files that run to thousands of lines, but those evolved over many iterations. Start simple.
