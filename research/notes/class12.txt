Diagramming with AI: From Data to Visual Structure
One of the most common tasks in research communication is producing a diagram that captures structure: the flow of a simulation pipeline, the architecture of a control system, the dependencies between components of an experimental setup. These diagrams appear in papers, presentations, posters, and documentation. They are also, for most researchers, tedious to produce by hand and difficult to keep current as the underlying system evolves. That’s the kind of bookkeeping task where AI assistance pays off, because it can translate a structural description into a visual layout far faster than you can drag boxes and draw arrows in a GUI.

Live demo: a curriculum flow chart in three steps
To illustrate a more elaborate workflow, let me share an example that starts with messy, print-formatted data and ends with a nice block diagram, touching three different AI tools along the way.
The source material is a curriculum sheet for the BS degree in our department, published on the MAE website as an Excel file.[5] The spreadsheet is not organized as a database; it is organized for printing. Each semester occupies its own small table, with course names, credit hours, and space for students to add their grade, all on a single page, arranged for visual clarity on paper rather than for machine readability. Eight of these mini-tables tile across the sheet, one per semester. If you wanted to build a flow chart showing how courses connect across semesters, you would first need to wrestle the data into a logical structure; that restructuring step is itself a useful job for AI. So here’s what I did.
Step 1: restructure the data with Claude in Excel. I opened the curriculum sheet in Excel and used the Claude add-in to read the scattered tables and consolidate them into a single logical table: semesters as rows, with courses listed across each row. This is a pre-processing step in the context engineering sense: the raw spreadsheet is not in a form that an LLM can reason about effectively, so we convert it into clean, structured data before asking the model to do anything creative with it. The output is a simple table that any human or machine can parse at a glance.
Step 2: prompt Claude to generate a block diagram. With the restructured data in hand, I opened Claude and prompted it to produce a curriculum flow chart: semesters arranged left to right, and courses as blocks. In fact, I asked Claude itself to make the prompt for me, in a follow-up question (still in Excel):
On the basis of the re-organized data, craft a prompt that I can use in another Claude thread in the app to request an Excalidraw diagram where each course will be shown as a block to the right of the corresponding semester.
Claude made the prompt, and I started a new thread on the desktop app. There, it generated the diagram using Excalidraw, rendering it directly in the conversation as an interactive visual that you can zoom, pan, and inspect without leaving the chat. 
Step 3: export and refine. The rendered diagram is not a static image; it is a live Excalidraw canvas. You can click into it, move elements, adjust labels, change colors, or add annotations. If the layout needs refinement, you can either edit manually in the canvas or ask Claude to revise the diagram in a follow-up prompt. When you are satisfied, you export to PNG for a presentation, to SVG for a publication, or to the native .excalidraw format for future editing.
To try another AI provider, I installed the Excalidraw MCP in Codex, by OpenAI. I used the same prompt as in Claude, and it generated a similar-looking diagram. Now I added an extra step: I asked for arrows indicating prerequisite chains. To get the pre-requisite information (which is not listed on the original curriculum sheets in Excel), I went to the course bulletin[6] on the agentic browser Comet (by Perplexity), and asked it to read the webpage and give me a list of courses with pre-requisites, which it promptly did. Here’s where the editable nature of the diagram proved useful: some prerequisite arrows were routed awkwardly, and I could fix them by hand.

Excalidraw: the tool behind the aesthetic
If you have read AI and technical blog posts in the last two years, you have seen Excalidraw's visual signature: a hand-drawn-but-polished look that splits the difference between a whiteboard sketch and a formal diagram. That aesthetic has become the default for architecture diagrams, system overviews, and conceptual illustrations across the AI engineering community, and Excalidraw is, overwhelmingly, how those diagrams are made.
Several features make it a natural fit for research workflows. Excalidraw is free and open source (MIT license), so there are no subscription costs and no risk of vendor lock-in. It runs entirely in the browser, with no installation and no account required: you open the page and start drawing.[7] It supports real-time collaboration: multiple users can edit the same canvas simultaneously via a shared link, which is useful for group meetings or co-advising sessions where you are sketching a system together. The underlying file format is text-based JSON, which means .excalidraw files can be versioned in Git, diffed meaningfully, and reviewed in pull requests alongside the code they document. There is a VS Code extension that renders Excalidraw canvases directly in the editor, so you can keep diagrams next to your source code without switching applications. And, as you just saw, Excalidraw now integrates with Claude and other AI assistants via MCP: the Excalidraw MCP server (released in February 2026) exposes tools for creating, querying, and editing diagram elements, which means you can ask an AI to draw on an Excalidraw canvas the same way you would ask it to write code in a file.[8] The MCP integration is what makes the workflow above possible: Claude is not generating an image; it is writing structured element definitions that Excalidraw renders as an editable diagram.
The hand-drawn aesthetic is more than cosmetic. In research contexts, it signals to readers that a diagram is a conceptual sketch rather than a precise engineering drawing: an important distinction when you are communicating system architecture, workflow logic, or abstract relationships rather than dimensioned geometry. When you need precision, you use CAD; when you need clarity about structure, Excalidraw is often the better choice.
Complementary diagramming tools
Excalidraw is excellent for freeform conceptual diagrams, but research communication involves several other diagramming idioms—flowcharts with strict semantics, system models, quick whiteboard-to-code sketches—and each has a tool particularly well suited to it.
Mermaid is a text-based diagramming language: you write a short block of structured syntax, and a renderer produces a flowchart, sequence diagram, state machine, entity-relationship diagram, Gantt chart, or class diagram.[9] The syntax is compact enough that LLMs generate it reliably, in fact, more reliably than they generate spatial layouts for visual editors. A typical Mermaid block is five to fifteen lines of plain text, which means it fits naturally into Markdown documents, renders natively in GitHub READMEs, Obsidian notes, Jupyter notebooks, and VS Code previews, and can be versioned alongside prose without any binary files. Mermaid Chart now offers an official Claude MCP connector for syntax validation and SVG rendering. The workflow pattern is simple: describe the diagram you want in natural language, let the LLM generate Mermaid syntax, preview it instantly, and refine through conversation. For engineering researchers, Mermaid is particularly useful for flowcharts describing simulation pipelines, state machines modeling control logic, and sequence diagrams documenting API interactions. It is free and open source (MIT license). The limitation is that you have less control over spatial layout than in a canvas-based tool: the renderer decides where to place nodes, and the results are sometimes awkward for large or densely connected diagrams.
draw.io (also known as diagrams.net) is the most widely used free diagramming tool in engineering and has been for years. It runs in the browser, requires no account, and stores files locally or in Google Drive, GitHub, or OneDrive. What makes it newly relevant for this class is its Generate feature, added in late 2025: you click a button, type a natural-language description, and draw.io produces a diagram using an LLM backend of your choice—Gemini, Claude, or ChatGPT.[10] The output is a native draw.io diagram with full access to the application's extensive shape libraries, including UML, BPMN, AWS/Azure architecture icons, electrical schematics, and network topologies. This makes draw.io the right tool when you need domain-specific shapes that Excalidraw and Mermaid do not provide: if your diagram includes standard engineering symbols, process-flow notation, or cloud infrastructure components, draw.io's library will have them. It is free and open source (Apache 2.0 license), and a draw.io MCP server exists for programmatic integration. The main trade-off relative to Excalidraw is aesthetic: draw.io produces clean, formal diagrams rather than the hand-drawn look, which is often what you want for publications but less appealing for blog posts and informal documentation.
tldraw takes a different approach. It is an open-source whiteboard with a feature called Make Real: you sketch a rough interface or diagram by hand on the canvas, and tldraw uses a vision model to convert your sketch into working HTML, CSS, and JavaScript.[11] The research application is narrower than the other tools—it is most useful for rapidly prototyping interactive visualizations, dashboards, or web-based explanations of research concepts. You sketch what you want the output to look like; the AI builds it. The underlying metaphor is powerful: instead of describing a visualization in words, you draw it, and the model interprets your spatial intent. tldraw is free and open source (Apache 2.0), raised $10 million in Series A funding in 2025, and requires an OpenAI API key for the Make Real feature. It is worth knowing about as a creative tool even if you do not use it regularly.
Napkin AI occupies yet another niche: you paste text—an abstract, a methods section, a set of bullet points—and it generates multiple visual representations across more than thirty diagram types, including flowcharts, timelines, comparison tables, mind maps, and process diagrams.[12] You click a spark icon next to any paragraph, and Napkin produces several layout options instantly. It is not a precision tool; you would not use it for a publication figure. But for quickly exploring how to visualize a concept, or for generating a first draft of a conference-talk slide, it is remarkably fast. Napkin AI has a free tier (500 credits per week, watermarked exports) and a paid tier at $9 per month.
Each of these tools implements the same underlying pattern: a human provides structural intent (in words, in syntax, or in a sketch), and an AI translates that intent into a visual artifact that the human can then refine. The differences are in input modality (natural language, structured syntax, freehand drawing, pasted prose), output formality (hand-drawn, schematic, interactive), and ecosystem integration (which file formats, which collaboration features, which shape libraries). Choosing among them is a context engineering decision in miniature: you are matching the tool's strengths to the specific communication task, the same way you match a model's capacity to a task's complexity when you practice budget forcing across model tiers.
Slide creation: from Markdown to a GW-branded deck
We now turn to how AI support is transforming the process of building technical slide decks, and we will see the same pattern as in the diagramming tools: a human specifies structural and visual intent, an AI handles the mechanical translation, and the output is a text-based artifact that can be versioned, edited, and reused.
Opening demo: building a GW-branded theme for Marp
The demo I built uses Marp, a Markdown-based presentation framework with a VS Code extension that renders a live preview of your slides as you edit.[13] Marp is free and open source, supports LaTeX math natively through MathJax, and lets you write an entire slide deck as a single plain-text Markdown file. Before we look at the anatomy of a Marp deck, let me tell you how I built the GW-branded theme that this deck uses, because the process is itself an illustration of patterns we have discussed in recent weeks.
The starting goal was simple: I wanted Marp slides that looked like they belonged to GW, not like the generic Marp default. The GW Division of External Relations maintains a refreshed brand-identity guide on its public communications website,[14] with a specified color palette, typography stack, and a set of conventions for graphic elements. Rather than manually transcribing any of that, I used the Claude Chrome extension[15] and asked it to read the GW brand pages directly, paying particular attention to the color palette, typography, and graphic-element conventions, and to summarize the specifications into a Markdown file.
From there the workflow moved into VS Code. I made a small sample two-slide deck in Marp, then asked Claude (via the VS Code extension) to generate a theme CSS that honored the GW specifications described in the Markdown file. It proceeded to generate the full styling sheet using the color palette for headings and accents, and appropriate treatment for title slides, section dividers, code blocks, and dark-mode slides. Since the typography is proprietary, Claude found close matches in free Google fonts. I then extended the sample deck to apply the features a typical technical talk needs: a title slide, a content slide with an equation, an image-from-URL slide, a section divider, a code block with syntax highlighting, a dark-mode slide, a blockquote slide, and an "impact" slide for a quotation or emphasis. With the Marp previewer open next to my editor, I iterated with Claude on the styling to improve it. When heading weights looked thin, I said so; when pagination was misplaced, I asked for it to move; when I wanted better contrast in the dark slide, Claude adjusted the relevant CSS variables: in VS Code, Claude edited the CSS file directly, asking for my approval each time. The iteration loop was fast because the preview renders instantly and because the feedback I was giving was visual and specific.

The Markdown summary is a pre-processed, structured artifact of the kind we emphasized in Class 10: clean, indexed context that Claude can reason about efficiently, not a noisy web page with navigation chrome and marketing copy. The CSS theme is a persistent semantic artifact: once built, it can be reused across every slide deck, and shared with anyone who wants a GW-branded starting point. (I will post both the theme CSS and the sample deck to the course repository.)
The anatomy of a Marp deck
A Marp deck is a single Markdown file. At the top is a short YAML front matter that tells Marp which theme to use, which math engine to enable, and any global options: in our deck it simply specifies marp: true, theme: gwu, and math: mathjax. Slides are separated by a horizontal rule, three hyphens on their own line (---), which is also standard Markdown. Inside each slide, the body is ordinary Markdown: headings with #, ##, ###; bullet lists; blockquotes with >; fenced code blocks with triple backticks and a language tag that triggers syntax highlighting; and inline or display LaTeX math between single or double dollar signs. Slide-level styling is controlled with HTML-comment directives: <!-- _class: title --> applies a class to the current slide only, selecting a layout defined in the theme CSS, while <!-- paginate: true --> is a global directive that turns page numbers on from that slide onward. Background images, sized and positioned, are written with an extended image syntax such as ![bg right:33% height:3cm](logos/gw_monogram.png), which tells Marp to place the image as a background element occupying the right third of the slide.
Two points are worth stressing. The first is that a Marp deck is entirely plain text: no binary .pptx blobs, no proprietary format, no "untitled.key" files. The second is that the styling is entirely external: the same Markdown source will render differently depending on which theme CSS is active. These two properties produce features that matter for research work.
Features that matter for research
LaTeX equation rendering is Marp’s first winning feature. It compiles MathJax or KaTeX directly into the rendered HTML, so expressions like the Schrödinger equation on slide 2 of the sample deck render exactly as they would in a LaTeX document or a journal PDF. This alone rules in Marp for technical presentations in a way that rules out most commercial AI slide builders, which either render math as images, mangle multi-line derivations, or decline to handle equations at all.
The second feature follows from the plain-text representation: a Marp deck is versionable. You can commit it to a Git repository alongside your manuscript, your simulation code, and your data-analysis notebooks; you can diff two versions meaningfully; you can review a collaborator's proposed slide edits as a pull request. The binary-file tax that has attached itself to PowerPoint for three decades—opaque diffs, merge conflicts no human can resolve, file-format breakage between versions—is gone. The same logic applies to the theme: the GW theme is a CSS file in the repository, not a set of manually-adjusted master slides that have to be copied deck-to-deck.
Third, the VS Code integration closes the loop with your AI assistant. The Marp extension provides a live preview in a side panel, so you can ask Claude to make a change, save the file, and see the result in under a second. Export to PDF, to standalone HTML, and even to .pptx for colleagues who insist on it, is available from a single command.

Alternatives: Gemini Canvas, Claude Code, and Gamma
Marp is not the only way to produce AI-assisted slides, and for some use cases it is not the right choice. A few alternatives worth knowing about: Gemini in Canvas mode can generate slide decks directly as HTML, with an "Export to Slides" button that sends the output to Google Slides. A practical constraint is that Canvas generates only about a dozen slides per turn, so long decks require you to work section by section and stitch results together. Claude Code can produce slide decks by writing HTML and CSS directly in your working directory, giving you full control over layout, animation, and any JavaScript you want to include for interactive elements. This is the most flexible option but also the one with the steepest setup cost, since you are effectively asking Claude to author a small website. 
Gamma.ai occupies the other end of the spectrum: it is an AI slide builder aimed at marketing-style decks, with strong defaults for visual polish, image generation, and modern layouts. It lacks a serious equation renderer, which makes it a poor fit for engineering content, though worth knowing about for non-technical audiences such as outreach talks, broader-impacts material, or NSF-style proposal summaries.[16]
A pattern is emerging across all three of these alternatives and Marp itself: researchers and technical communicators are increasingly ditching PowerPoint and its equivalents in favor of what some have started calling "HTML-maxing," the move to slide decks that are, under the hood, just HTML, CSS, and a little bit of JavaScript.[17] The reasons are the same ones that made Marp appealing: HTML is versionable, AI-generatable, portable across devices and operating systems, trivially embeddable in a web page, and fully customizable by anyone who can read a CSS file. For a generation of researchers who already live in Markdown and the browser, the .pptx file increasingly looks like the .doc file did a decade ago: a heavy, proprietary format that survives mostly out of habit.
Note-Taking and the Personal Knowledge Base
We began this session with a promise: that AI tools for organizing research knowledge are not just convenient utilities but implementations of a vision that has been gestating for eighty years. The diagramming tools turned structural descriptions into visual artifacts; the slide tools turned prose into presentations. Both externalized knowledge that would otherwise remain locked in a researcher's head or scattered across disconnected files. But neither addresses the deeper problem that Vannevar Bush identified in 1945, and that every PhD student eventually confronts: how do you organize the accumulating mass of things you have read, thought, and discovered so that your future self can actually find, connect, and build on them?
Bush called his imagined solution the Memex: a mechanized desk with translucent screens, microfilm storage, and a system of levers and buttons that would let a researcher retrieve several articles or pictures on one screen, write comments that could be stored and recalled together, and create links between related items, mapping the thought process and path of each user and saving it for others to experience.[18] The Memex was never built, but the underlying insight endures: the value of a knowledge store lies not in the documents themselves but in the trails between them: the connections, annotations, and cross-references that reflect how a particular mind made sense of a particular body of material. Those trails are what distinguish a personal knowledge base from a filing cabinet.
For decades, the tools available to researchers fell short of Bush's vision in a specific and frustrating way. You could store documents (filing cabinets, then folders on a hard drive, then cloud storage). You could search them (grep, then Spotlight, then Google Drive). But the connective tissue—the annotations, the cross-references, the evolving synthesis of what you know—had to be maintained by hand. And humans, as a rule, do not maintain it. The initial enthusiasm for note-taking fades; the cross-references go stale; the carefully organized folder structure drifts into a miscellaneous pile. The maintenance burden grows faster than the value, and so researchers abandon their systems and retreat to the only knowledge base that persists: their own memory, which is unreliable, unsearchable, and mortal.
This is where AI changes the equation, but by handling the bookkeeping that makes a knowledge base worth maintaining. The tools we examine next attack this problem at two levels: first, a note-taking application designed for the kind of linked, associative knowledge that Bush envisioned; second, a recently proposed pattern that uses an LLM to do the maintenance work that humans inevitably abandon.
Obsidian: local Markdown as a knowledge substrate
Obsidian is a note-taking application built on a simple but consequential design decision: every note is a plain Markdown file stored on your local filesystem.[19] There is no proprietary database, no cloud dependency, no vendor-controlled format. Your notes are text files in a folder. You can open them in any text editor, version them in Git, back them up however you like, and if Obsidian the company disappears tomorrow, your knowledge remains exactly where it was.
This matters for research in ways that are not immediately obvious. Markdown is the format that LLMs read and write most naturally; a folder of .md files is a context surface that any AI agent can operate on without format conversion. Git compatibility means your knowledge base has full version history, branching, and diffing for free. Local storage means you never send unpublished research to a third-party server unless you choose to. And the simplicity of the format means the system can grow for years without the performance degradation that plagues database-backed applications at scale.
What makes Obsidian more than a text editor is its support for wiki-style links: you type [[concept name]] in any note, and Obsidian creates a clickable link to a note with that title, creating the note if it does not yet exist. This is Bush's associative trail, implemented as a two-keystroke gesture. Over time, as you link notes to one another, a network of connections emerges—visible in Obsidian's graph view, which renders every note as a node and every link as an edge, letting you see the shape of your knowledge at a glance. Hub nodes are the concepts that connect many ideas; orphan nodes are the ideas that have not yet been integrated. The graph is a diagnostic tool for your own understanding.
Obsidian's plugin ecosystem (2,700+ community plugins as of April 2026) extends the core application in directions that matter for research workflows. Copilot for Obsidian, the most-downloaded AI plugin with over 100,000 users, adds a chat interface that can query your vault using any LLM provider: OpenAI, Anthropic, Google, or local models via Ollama for fully offline operation.[20] Smart Connections uses local embeddings to surface semantically related notes as you write, creating a second layer of linking that operates on meaning rather than explicit references.[21] The Dataview plugin treats YAML frontmatter in your notes as a queryable database, so you can generate dynamic tables and lists (e.g.,"show me all notes tagged #CFD that cite papers from 2025") without leaving your vault. And the Obsidian Web Clipper browser extension converts any web article to a Markdown file in your vault with one click, which is how most Obsidian users get source material into their system in the first place.
For engineering researchers, Obsidian's combination of local storage, Markdown format, AI plugin support, and Git compatibility makes it the most natural home for a personal knowledge base. It may not be the flashiest tool available for note-taking, but it imposes the fewest constraints on how you organize your thinking, and every file in your vault is immediately accessible to any AI agent you point at the folder.
The Karpathy Wiki: LLM-maintained knowledge compilation
On April 4, 2026—just over two weeks ago—Andrej Karpathy published a short document on GitHub describing a pattern for building personal knowledge bases using LLMs.[22] The document, titled simply "LLM Wiki," accumulated over 5,000 stars and 4,000 forks in its first two weeks, spawned dozens of open-source implementations, and generated hundreds of comments debating its merits and limitations. It is, by any measure, the most viral idea in personal knowledge management this year.
The proposal is simple enough to state in a single paragraph. In a RAG-based knowledge base, you upload documents and have the model retrieve relevant chunks at query time, rediscovering knowledge from scratch on every question. Instead of using RAG, the idea is to have the LLM incrementally build and maintain a persistent wiki: a structured, interlinked collection of Markdown files that sits between you and your raw sources. When you add a new source, the LLM does not just index it for later retrieval; it reads it, extracts the key information, and integrates it into the existing wiki, updating entity pages, revising topic summaries, noting where new data contradicts old claims, and strengthening the evolving synthesis. The knowledge is compiled once and then kept current, not re-derived on every query.
Karpathy's own framing captures the idea crisply: "Obsidian is the IDE; the LLM is the programmer; the wiki is the codebase." Engineers understand codebases. A codebase is compiled from source, kept current through edits, version-controlled, internally linked, and grows in value as it grows in size. A wiki maintained by an LLM works the same way, except the compiler is the model and the source files are whatever you feed it: papers, articles, notes, transcripts, images.
Three-layer architecture
The architecture has three layers, and each of them maps onto concepts you already know:
Raw sources—a folder of immutable source documents (articles, papers, data files, images). The LLM reads from this layer but never modifies it. This is your source of truth. In the vocabulary of Class 10: this is static context with clear provenance.
The wiki—a directory of LLM-generated Markdown files: summaries, entity pages, concept pages, comparisons, an overview, a synthesis. The LLM owns this layer entirely: it creates pages, updates them when new sources arrive, maintains cross-references, and keeps everything consistent. You read it; the LLM writes it. In the vocabulary of context engineering: this is compiled and cached context—the "pre-process" principle from the five agent design principles, applied at the scale of an entire knowledge domain rather than a single prompt.
The schema—a configuration document that tells the LLM how the wiki is structured, what the conventions are, and what workflows to follow when ingesting sources, answering questions, or maintaining the wiki. Karpathy explicitly names this: for Claude Code, it is a CLAUDE.md file; for OpenAI Codex, it is an AGENTS.md file. This is the CLAUDE.md pattern from Class 7 and Assignment 4, scaled up to govern a long-lived knowledge system. If you wrote a thoughtful CLAUDE.md for your research project in Assignment 4, you are three-quarters of the way to running your own LLM-wiki knowledge base.
Three operations
The lifecycle of the wiki has three verbs:
Ingest. You drop a new source into the raw folder and tell the LLM to process it. The model reads the source, discusses key takeaways with you, writes a summary page in the wiki, updates the index, touches ten to fifteen relevant existing pages to integrate the new material, and appends an entry to the log. You supervise and guide; the LLM does the bookkeeping. A single source might ripple through a dozen wiki pages, the kind of cross-referencing work that humans start strong on and then abandon by month three.
Query. You ask questions against the wiki. The LLM reads the index first to find relevant pages, drills into the content, and synthesizes an answer with citations. Answers can take different forms: a Markdown page, a comparison table, a slide deck via Marp, a chart via Matplotlib. The critical insight is that good answers get filed back into the wiki as new pages. A comparison you asked for, an analysis you ran, a connection you discovered: these are valuable and should not disappear into chat history. Your explorations compound in the knowledge base just as ingested sources do.
Lint.[23] Periodically, you ask the LLM to health-check the wiki: find contradictions between pages, flag stale claims that newer sources have superseded, identify orphan pages with no inbound links, surface important concepts that are mentioned but lack their own page, and suggest data gaps that could be filled with a web search. This is the maintenance loop that humans invariably abandon; the LLM does not get bored and does not forget to update a cross-reference.
Two special files keep the system navigable. An index.md file is a content-oriented catalog: every page listed with a link, a one-line summary, and metadata. The LLM updates it on every ingest and reads it first when answering queries. At moderate scale (a few hundred pages), this avoids the need for embedding-based search infrastructure entirely. A log.md file is a chronological, append-only record of what happened and when: ingests, queries, lint passes. It is the wiki's episodic memory, providing a timeline of the knowledge base's evolution.
Why this matters for our course
The LLM wiki fits this class well because it retroactively unifies several things we have studied this semester. Consider how many prior-class concepts it instantiates simultaneously:
The schema is a CLAUDE.md (Class 7). The operations are structured as SKILL.md files—several community implementations organize them exactly this way, with separate skill files for ingest, query, and lint (Classes 8 and 9). The index.md is literal indexing; the ingest operation is pre-processing context for consumption; the wiki itself is cached stable context; the schema scopes the agent's behavior to the relevant domain; and log.md provides instrumentation (Class 10's five agent design principles, all five accounted for). The wiki is externalized semantic memory; the log is episodic memory (Class 10's memory taxonomy). And the whole system is a context-engineering artifact designed to make every future query cheaper and more accurate by doing the expensive synthesis work once, at ingest time, rather than rediscovering it from scratch on every question.
In other words: the LLM wiki is what you get when you take the principles of this course seriously and apply them to the oldest problem in research: keeping track of what you know.
Some honest caveats
The Karpathy gist has attracted both enthusiastic adoption and substantive criticism, and you should leave this class with a calibrated understanding rather than uncritical excitement.
Scale is unproven. Karpathy explicitly describes a system at moderate scale: roughly 100 sources, a few hundred wiki pages. At that size, the index-based navigation works well and the LLM can maintain consistency. At 10,000 pages, Markdown is not a database, and LLM-maintained cross-reference consistency will almost certainly degrade. Several commenters on the gist have already identified this ceiling and proposed hybrid solutions involving graph databases or structured search engines like qmd.[24]
Hallucination feedback loops are a real risk. Unlike RAG, where the model always retrieves from original source text, the LLM wiki lets the model reason over its own prior outputs: the summary pages, the entity pages, the synthesis. If a summary introduces a subtle distortion, that distortion can propagate through subsequent updates. The mitigation is disciplined provenance: every wiki page should cite the raw sources it draws from, and periodic re-ingestion from raw sources can correct drift. But the risk is structural and worth understanding.
It is not a wiki in the traditional sense. Several commenters correctly note that a wiki, properly defined, is collaborative and multi-user. Karpathy's proposal is a single-user, LLM-maintained knowledge base. "LLM-compiled knowledge base" would be more precise, but "LLM Wiki" is what went viral and what people will search for. Acknowledge the terminological imprecision without letting it obscure the genuinely useful pattern underneath.
The maintenance problem has not been solved; it has been displaced. The LLM handles the bookkeeping, but the human still needs to curate sources, review the LLM's integrations, and run lint passes. If you dump a hundred papers in and never check what the LLM wrote, you are trusting a probabilistic system to build your understanding for you, which leads to an "illusion of knowledge." The LLM is doing the mechanical labor; the intellectual judgment remains yours.
These caveats are features of the discussion, not reasons to dismiss the pattern. They mean that researchers who adopt it will do so with clear eyes about what it can and cannot do, which is the disposition this course has been cultivating all semester.
The Memex, eighty-one years later
Let’s return, finally, to Vannevar Bush. What he imagined in 1945 was not a search engine; it was a system where the connections between documents—the trails of association that a particular researcher builds while thinking through a problem—would be as durable and retrievable as the documents themselves. The web gave us the documents and the links, but not the personal trails; search engines gave us retrieval, but not accumulation; RAG gave us grounded answers, but not persistent synthesis. The LLM wiki, for all its rough edges, is the first pattern that credibly addresses all three: the documents are preserved as immutable raw sources, the trails are maintained as interlinked wiki pages, and the synthesis compounds with every source you add and every question you ask.
Bush wrote that the Memex would let a researcher build a trail through a body of literature and save it for others to experience. The LLM wiki does not yet do that; it is personal, single-user, and early-stage. But the direction is unmistakable, and the tools are available today: Obsidian for the substrate, Claude Code or Codex for the agent, a CLAUDE.md for the schema, and a folder of Markdown files for the compiled knowledge. You already know how to write the schema; you already understand why pre-processing matters, why indexing beats brute-force retrieval, and why persistent artifacts are more valuable than ephemeral chat responses. The conceptual infrastructure is in place. The challenge to you is whether you will use it to help you learn!
